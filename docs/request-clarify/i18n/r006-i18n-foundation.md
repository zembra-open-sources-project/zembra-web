# r006-i18n-foundation

日期：2026-05-15

## 需求背景

项目正在为国际化推广做准备。当前 Web 前端已经存在中文和英文混排的 UI 文案，文案分散在首页、同步设置页、主题切换、全局 toast 和测试用例中。为了后续面向不同语言用户推广，需要先建立统一的 i18n 底座和开发执行规范。

## 已确认需求

| 项目 | 结论 |
| --- | --- |
| 技术组合 | 使用 `i18next + react-i18next` |
| 首批语言 | 简体中文、繁体中文、英语 |
| 默认展示语言 | 简体中文 |
| 默认回退语言 | 英语（`en-US`） |
| 首批改造对象 | 前端 UI 壳文案、可访问性文案、表单提示、按钮文案、状态文案、日期格式 |
| 语言偏好 | 浏览器本地持久化，刷新后保留 |
| 开发方式 | 按项目流程先落需求澄清、设计文档和执行计划，经审核后再实现 |

## 范围边界

### In Scope

- 新增前端 i18n 基础设施。
- 新增简体中文、繁体中文、英语三套语言资源。
- 首页 UI 文案迁移到翻译资源。
- 同步设置页 UI 文案迁移到翻译资源。
- 全局 toast、主题切换等应用壳文案迁移到翻译资源。
- 日期展示根据当前语言选择对应 `Intl.DateTimeFormat` locale。
- 在设置页新增语言选择入口，便于用户统一管理语言偏好。
- 更新相关单元测试，避免测试绑定单一裸文案。
- 补充开发约束，避免后续新增 UI 裸文案。

### Out of Scope

- 不翻译用户创建的笔记内容、标签名、Field 名。
- 不翻译后端原样返回的错误文本。
- 不引入服务端翻译、机器翻译或远程文案管理系统。
- 不在第一阶段引入 URL 语言前缀，例如 `/en/settings/sync`。
- 不处理 SEO 多语言站点能力。
- 不改变 API Client、Repository、数据库 schema 或 Supabase 同步协议。

## 关键约束

- 新增依赖必须符合 `docs/references/dependency-constraints.md` 的轻量依赖原则。
- i18n 能力只能服务于前端展示层，不让 UI 组件依赖数据库、SQLite、Supabase 或 schema 细节。
- 文案 key 需要稳定、可读，避免直接使用整句原文作为 key。
- 日期、数字和动态文本必须通过参数或 formatter 处理，禁止拼接造成语言顺序无法调整。
- 所有新增函数需要保留项目现有文档字符串风格。

## 待审核问题

- 语言选择入口放在设置页顶部工具区，与主题切换并列。
- 繁体中文初版建议使用人工维护的 `zh-TW` 资源，不做运行时简繁转换。
- 英语 locale 使用 `en-US`，同时作为缺失翻译时的默认回退语言；后续如需英式英语再扩展 `en-GB`。
