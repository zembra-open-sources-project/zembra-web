# r019-sync-status-feedback 需求澄清

日期：2026-06-22

## 背景

后端 Supabase 同步功能更新后，前端需要基于实时 OpenAPI 判断适配范围。该需求属于前端需求，所有影响用户感知的后端功能变化都必须有对应展示内容，但展示方式应降低用户认知负担，避免把后端内部同步阶段直接暴露成复杂操作模型。

## 需求理解

Web UI 需要继续把同步表达为一个用户心智中的 `Sync` 操作。后端提供的 `push`、`pull` 和 `run` 属于同步执行过程的能力边界，但前端不应强行把 `Push`、`Pull` 拆成独立按钮，因为用户感知上它们都属于同步，拆分会增加理解和误操作成本。同步状态展示只需要覆盖失败和错误信息，帮助用户理解同步是否出问题，不需要展示 cursor 明细或后端内部状态行。

## 已确认结论

| 项目 | 结论 |
| --- | --- |
| OpenAPI 来源 | `http://127.0.0.1:3000/api-docs/openapi.json` |
| 用户操作模型 | 前端只保留一个 `Sync` 同步入口 |
| `POST /sync/run` | 作为用户触发同步的主入口 |
| `POST /sync/push`、`POST /sync/pull` | 不作为独立用户操作入口展示，可作为后续内部实现能力评估 |
| `GET /sync/status` | 只展示失败和错误相关信息 |
| 状态展示边界 | 不展示 cursor rows、scope、last change id 等内部同步细节 |
| Settings 边界 | Supabase section 继续负责配置、测试和启用开关 |
| 首页边界 | 首页继续承载用户可见的手动 `Sync` 操作 |

## 范围

- 保留现有首页 `Sync` 入口，用户不需要区分 `Push` 和 `Pull`。
- 适配同步结果展示，保持用户看到的是一次同步操作的成功或失败反馈。
- 在 Supabase 设置区域或同步反馈区域展示同步失败和错误信息。
- 如果使用 `GET /sync/status`，仅提取失败和错误相关字段用于展示。
- 更新前端 API 类型、client 映射和测试，使同步状态错误展示有稳定数据来源。
- 更新英文、简体中文、繁体中文文案。

## 不做范围

- 不新增独立 `Push`、`Pull` 按钮。
- 不展示 sync cursor 明细、scope 列表、last change id 或内部同步状态表。
- 不引入 Supabase SDK。
- 不让 UI 组件直接依赖 Supabase 查询实现或数据库 schema。
- 不改后端 `/sync/*` 契约。

## 验收标准

- 用户在首页只看到一个清晰的 `Sync` 同步操作。
- 同步成功时展示一次同步的结果反馈。
- 同步失败时展示可读的失败或错误信息。
- Settings 中的 Supabase section 仍聚焦配置、测试和启用状态，不变成同步工作台。
- 前端测试覆盖 sync client 对状态错误字段的映射，以及 UI 对同步失败信息的展示。
